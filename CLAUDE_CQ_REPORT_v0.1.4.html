Critical Issues (human review: non-critical architecture problems)

Critical
Electron Sandbox Disabled
The application disables Electron sandbox and web security, allowing renderer process direct Node.js access.
src/main/index.ts (lines 65-66) webPreferences: { sandbox: false, // CRITICAL SECURITY ISSUE webSecurity: false // ADDITIONAL SECURITY ISSUE }
Impact: Renderer has full Node.js access. XSS vulnerability becomes full system compromise.
Human: Not sandboxed because there's weird stuff happening with vite to run the hot-reload dev server from within the app
Critical
Path Traversal Vulnerability in Asset Operations
File paths from renderer are not validated for directory escape attempts using ".." sequences.
src/main/fileSystem.ts (line 380) export function getAssetFilePath( projectPath: string, assetRelPath: string ): string { const assetsDir = path.join(projectPath, '_include', 'assets') return `file://${path.join(assetsDir, assetRelPath)}` }
Risk: Attacker can pass assetRelPath = "../../sensitive-data" to escape assets directory and access arbitrary files.
Human: Probably real, but you will be accessing paths on your computer from... your computer. There's no internet and you write the prototypes you run, so this is fine. Don't open untrusted projects.
Critical
Unvalidated IPC Handlers Expose File Operations
IPC handlers perform file operations without input validation or permission checks.
src/main/ipc.ts (lines 281-296) ipcMain.handle('assets:drop', async ( _, filePaths: string[], targetFolder?: string ) => { if (!currentProjectPath) return [] // ... proceeds with file operations without validation }
Issue: Handler accepts arbitrary paths. Combined with sandbox:false, allows unrestricted file system access.
Human: Again, if your own code wants to escape the sandbox, it can. But it's code you put there.
